**What:** 添加 LlamaIndex 核心依赖到 `pyproject.toml`

**Why:** 设计文档选择 LlamaIndex 负责向量索引和检索，但目前未安装

- 100+ 预构建 readers (PDF, YouTube, Notion 等)

设计文档决策：LlamaIndex 优先，ContentService 退化为文件 CRUD 层。
实现时需要创建 adapter 层，将 LlamaIndex 的 VectorStoreIndex 与现有存储层集成。

## [ENG-002] SSE Citation 协议扩展

**What:** 扩展 `markwritter/api/models/chat.py` 的 `ChatEvent` 模型，添加 citation 事件类型

**Why:** 第二意见指出当前 SSE 协议不支持流式返回引用，需要额外 API 调用

**Pros:**
- 单次请求完成回答 + 引用
- 更好的用户体验（引用随回答逐步显示）

**Cons:**
- 需要修改前端 SSE 解析逻辑
- 增加后端事件类型复杂度

**Context:**
当前 `ChatEvent` 只有 `thinking`, `text_delta`, `done`, `error` 四种类型。
需要添加 `citation` 类型，携带 `{file_path, page_num, text_snippet}`。

**Related:** #Outside Voice Issue #8

## [ENG-003] Path → ContentID 映射层

**What:** 创建 `markwritter/storage/path_resolver.py`，实现文件路径到 content_id 的映射

**Why:** `selected_sources` 使用文件路径，但向量索引使用 UUID-based content_id

**Pros:**
- 解耦路径和索引 ID
- 支持文件重命名后的索引保持

**Cons:**
- 增加一层抽象
- 需要维护映射缓存

**Context:**
第二意见指出：如果 `selected_sources` 包含文件路径但向量索引使用 UUIDs，需要 path→content_id 映射层。
可以用 `Content.source_path` 字段建立映射，但需要确保该字段被正确索引。

**Depends on:** —

**Related:** #Outside Voice Issue #2

## [ENG-004] LRI 缓存设计

**What:** 实现内存缓存层，缓存热门查询的向量检索结果

**Why:** 性能审查建议 MVP 就加缓存，减少向量检索延迟

**Pros:**
- 减少 50-200ms 的检索延迟
- 降低向量数据库负载

**Cons:**
- 需要设计缓存失效策略（vault 文件变化时）
- 内存占用

**Context:**
审查决策：MVP 加缓存。需要指定：
- Cache key 设计（query hash? session_id + query?）
- TTL policy
- Size limit
- Invalidation trigger

## [ENG-005] Watchdog 集成

**Why:** 设计文档选择"启动时全量扫描 + watchdog 增量更新"，但目前使用轮询

## [ENG-006] Conversation History 一致性

**What:** 明确 `ChatState.conversation_history` (内存) 和 `chat_sessions.conversation_history` (DB) 的同步策略

**Why:** 第二意见指出存在两个事实来源，可能导致不一致

**Pros:**
- 避免状态不一致
- 支持重试和并发请求

**Cons:**
- 增加 DB 写入频率
- 需要设计同步原语

**Context:**
建议策略：每个请求结束后异步更新 DB，内存状态作为单一事实来源。
